2. 项目背景与问题定义

公司当前业务仍主要分散在外部电商平台、ERP、表格、截图、私聊与人工补录之中，造成任务、商品、设计、采购、仓库与财务数据无法统一沉淀。系统建设目标不只是替代表格，而是从零建立公司内部可持续扩展的数字化底盘。

2.1 现状痛点

任务入口分散，信息口径不统一，责任人和补资料边界不清。

新品任务经常由运营先给链接和截图，设计出图、编码建档、信息维护不同步，最后由仓库兜底补资料，效率低且易错。

原品开发缺少对 ERP 历史产品、历史资产和旧版设计的标准化复用。

采购任务在业务上并不经过设计和生产，但当前缺少独立的直连路径。

成本价录入缺少统一模型，无法沉淀尺寸、面积、工艺、材料与价格规则。

导出、绩效、财务等后续管理需求没有统一的数据源与接口预留。

登录、权限、组长管理、部门边界、审计日志缺乏系统化设计。

2.2 建设目标

统一所有正式业务任务的创建、编码、建档、流转与结单。

将任务、SKU、分类、资产版本、成本价、仓库接收、采购信息沉淀为统一数据模型。

把设计、审核、外包、采购、生产从“串行主线”改造成“任务下的条件子流程”。

为导出、KPI、人事接口、财务统计和后续 API 扩展预留标准模块。

主题

定义

条件子流程

设计、审核、外包、采购、生产

统一要求

所有正式任务必须有任务号；所有商品必须有 SKU；所有关键动作留痕；所有数据可追溯、可导出、可对接

系统边界

3.1 设计原则

任务优先：任何模块都不能脱离任务独立流转。

前置建档：编码、SKU、分类、成本基础信息尽量前置，不把补资料责任推给仓库。

按任务类型分型：原品开发、新品开发、采购任务分别定义结单条件。

入口统一、链路可分：用户操作模型尽量一致，但允许底层按权限和网络策略分流。

模块预留：导出、API、绩效、财务、规则引擎一开始就留扩展位。

5.2 权限原则

数据权限至少支持：本人、组内、部门内、跨部门授权、全局可见。

动作权限与可见权限分离：能看不等于能改，能改不等于能导出。

导出、成本、财务、接口配置默认高敏感。

组长需要独立的“分配/催办/查看组统计/纠偏”权限。

所有关键权限变更与高风险动作必须进入审计日志。

6. 总体业务流程（重构后）

6.1 统一主线

创建任务

按规则自动生成 Task No、绑定或生成 SKU，并建立商品档案

入 ERP / 建档

维护商品必要信息（分类、规格、文案、成本等）

进入仓库节点，由采购/仓库部门承接

按任务类型判断是否满足结单条件

6.2 条件子流程

设计子流程：仅在原品开发和新品开发等需要设计的任务上触发。

审核子流程：仅在存在待审核设计版本时触发。

外包子流程：仅在审核判定需转定制/外包时触发。

采购子流程：在采购任务中为主支线，在其他任务中按需触发。

生产子流程：本期只预留状态位和接口位，不作为强制主线。

6.3 三类任务主路径

任务类型

主路径摘要

原品开发

从 ERP 已有品发起 → 自动带出已有 SKU 与分类 → 分配设计 → 新版本设计/审核 → 资料齐套 → 仓库接收 → 结单

新品开发

录入外部链接/截图/文案 → 生成新品 SKU 与任务号 → 尽早建档与维护基本信息 → 分配设计 → 审核通过 → 补齐资料与成本 → 仓库接收 → 结单

采购任务

录入可直接采购商品 → 生成任务号/SKU → 建档并维护采购价/成本价/商品必要信息 → 采购/仓库承接 → 无图无审核也可结单



7. 结单逻辑

结单必须从“单一流程结束”改为“主状态 + 子状态 + 关键字段齐套”的联合判断。系统应根据任务类型自动计算“不可结单原因”。

任务类型

必须满足的结单条件

原品开发

已绑定已有 SKU；设计已完成并有终稿版本；审核通过；仓库已接收；必要字段齐全；不存在未处理退回项

新品开发

已生成新品 SKU；任务已建档；设计已完成并审核通过；成本与商品基础信息齐全；仓库已接收；不存在未关闭异常

采购任务

已生成或绑定 SKU；已建档；采购价/成本价已维护；仓库已接收；商品必要信息齐全；允许无图、无审核结单



7.1 不可结单原因示例

缺少 SKU 或任务号。

需设计任务尚无终稿或审核未通过。

成本价未维护或商品规格不完整。

仓库节点尚未接收。

存在待补资料、待处理退回、待复核的异常项。

8. 功能需求

8.1 登录与组织管理

支持账号密码登录、后续单点登录对接预留。

支持组织架构、部门、组、岗位与角色模板配置。

支持超级管理员、部门管理员、组长三级管理面板。

支持账号停用、离职交接、角色变更、审计留痕。

8.2 工单中心

统一展示全部任务，支持按任务类型、部门、负责人、状态、SKU、分类、时间筛选。

支持看板与列表双视图。

支持快速查看任务主状态、设计子状态、仓库状态、缺失字段提示。

支持批量分配、批量催办、批量导出（预留）。

8.3 任务创建与建档

创建时必须选择任务类型：原品开发 / 新品开发 / 采购任务。

原品开发支持从 ERP 已有品中按 50+ 分类筛选检索。

新品开发支持录入外部链接、截图、文案、规格与说明。

采购任务支持录入采购相关字段并直接走采购/仓库路径。

正式提交时自动生成 Task No，并绑定/生成 SKU。

提交后自动推动 ERP/主数据建档。

8.4 商品信息与成本维护

支持维护产品名称、分类、规格、文案、备注、供应信息、仓库必要字段。

必须预留成本价预填能力：支持按分类、尺寸、面积、单价、材料、工艺等规则自动计算或预填成本。

成本价规则需做成可配置规则模型，后续可由业务自行维护。

采购价与公司内部成本价在采购任务中需支持统一口径输出。

8.5 设计资产中心

仅对需要设计的任务启用。

支持参考图、原图、初稿、修改稿、终稿等版本上传。

支持 Asset No、版本号、命名校验、重复文件校验、历史版本查看。

上传能力应支持大文件、断点续传和后续 NAS/上传服务扩展。

8.6 审核、交班与外包

审核仅在待审核设计版本存在时触发。

支持通过、打回、转派、交班、关闭、转外包。

交班需保留交班原因、当前结论、风险点、待跟进事项。

外包需支持外包单、交付包、回传复核与再次回退。

8.7 采购/仓库节点

采购与仓库在同部门视角下需要统一承接面板。

采购任务可直接进入该节点，无需设计与审核。

仓库对资料不齐任务可退回，并明确退回原因。

仓库接收后可进入完成/打包/待结单。

8.8 导出、开放接口、KPI 与财务预留

系统需从第一版开始预留导出中心：工单数据导出、仓库数据导出、后续自定义模板导出。

预留标准 API 层，供后续人事 KPI 考核、ERP、BI、财务或其他系统接入。

预留 KPI 指标口径：任务量、按时率、打回率、设计产出、审核效率、仓库承接效率等。

预留财务口径：销售额、成本、毛利、任务成本、分类成本统计等，先完成字段沉淀与接口位，不在本期做完整财务系统。

14. 验收标准

系统能够按“原品开发 / 新品开发 / 采购任务”三类任务正式创建并流转。

所有正式任务均能生成 Task No，且拥有有效 SKU 关系。

新品任务不再默认由仓库兜底补核心资料，系统可在主线中提前校验缺失项。

采购任务可在无设计、无审核的情况下直接进入采购/仓库路径并结单。

成本价字段与规则入口已预留，可支持后续按尺寸/面积/材料计算。

导出中心、开放 API、KPI、财务口径均有明确模块和接口预留。

权限体系支持超级管理员、部门管理员、组长和业务角色分层管理。

系统日志可追踪任务创建、状态变化、导出、权限变更与关键操作。
